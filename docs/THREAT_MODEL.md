# cblicense threat model

**Read this before shipping.** cblicense is a small piece of crypto inside a much bigger system; it can only help if you understand what it does and doesn't do.

## What cblicense v0.1 defends against

| Threat | How v0.1 helps |
|--------|----------------|
| Someone re-flashing the firmware to another device of the same model | Codes are bound to a SHA-256 of the device's hardware fingerprint (efuse MAC + chip rev on ESP32; eth0 MAC + Pi serial + machine-id on Linux). The code minted for device A will fail to verify on device B. |
| Casual code sharing in a forum / Discord | A leaked code only activates the device whose fingerprint it was minted for. |
| Cross-product reuse | The product `family` byte is mixed into the HMAC. A code minted for the controller won't validate on the PLC. |
| Casual code-tampering | Single-bit flips in the typed code fail the HMAC compare. |
| Timing-based key recovery | All compares are constant-time. |

## What cblicense v0.1 does NOT defend against

This is the important section. **Be honest with yourself about which of these matters for your product.**

### 1. Salt extraction → global compromise

cblicense v0.1 is **symmetric**. The 32-byte vendor salt that mints codes is the same value baked into firmware to verify them. Whoever can read the firmware can mint codes for any device in the family forever.

How this happens in practice:
- Customer with `esptool.py read_flash` dumps the ESP32 image.
- Linux/Pi side: `sudo strings /opt/evse/controllerd | grep ...`, or just `gdb -p`.
- A disassembler walks straight to the salt buffer (high-entropy, easy to find).
- Reverse-engineering forum posts the salt; clones now mint their own codes.

**Mitigation:** ship Ed25519 signed-license files (v0.2). The device only carries the *public* key, which can be in a billboard. The mint key never leaves your CI server. Reverse-engineering a device gives the attacker the public key, which is useless for forgery.

**Until v0.2 ships:** treat v0.1 as casual-piracy prevention only — appropriate for licensing fees on the order of "an annoyance to bypass" but not for revenue-critical features.

### 2. MAC spoofing on Linux

The Pi's MAC is fully writable: `sudo ip link set dev eth0 address aa:bb:cc:dd:ee:ff`. An attacker who clones the SD card image and changes the MAC to match the original device can replay a leaked code.

**Mitigation already partially in place:** the Linux fingerprint provider mixes the MAC with `/etc/machine-id` (regenerated by `systemd-firstboot --setup-machine-id` on a fresh image — varies per Pi unless the attacker also clones `/etc`) and the BCM2711 Pi serial from `/proc/cpuinfo` (read directly from silicon, not spoofable from userspace).

**Stronger mitigation (planned v0.5):** burn a per-device secret into Pi OTP (not user-readable) at provisioning time, and HMAC over that secret.

### 3. ESP32 efuse MAC override

The ESP32 efuse MAC is normally read-only after burn. *But* the WiFi MAC seen from `WiFi.macAddress()` can be overridden at runtime via `esp_wifi_set_mac()`. cblicense's ESP32 provider reads `esp_efuse_mac_get_default()` directly (not the WiFi-stack-overridable copy), which is harder to fake without burning the efuse. Burning the efuse to a forged value is possible on dev boards but bricks them for that forgery — not zero-cost for an attacker.

**Stronger mitigation (planned v0.5):** burn a 256-bit per-device secret into a custom efuse block at provisioning, and HMAC over that. Reading the secret out via JTAG is blocked by the efuse read-protect bit.

### 4. Replay across firmware versions

Today's code is `HMAC(salt, "cblicense\x00v1" || family || device_id)`. If the salt remains the same and only the firmware version changes, codes minted for one version will validate on others. That's intentional: a license shouldn't expire because of an OTA update.

**If you want per-version codes:** change the salt with each major firmware release, OR move to v0.2 signed licenses where you can carry an explicit `min_version` field.

### 5. No expiry, no feature flags, no revocation

v0.1 short codes are perpetual and carry no payload other than family + device binding. To support tiered SKUs ("basic" vs "pro"), time-limited demos, or per-device revocation, you need v0.2 (Ed25519 signed-license files), which carries:
```json
{ "device_id": "...", "expiry": "2027-12-31", "features": ["pro", "v2g"], "nonce": "...", "signature": "..." }
```

### 6. Determined kernel-mode attacker

If the attacker can patch your binary, they can rewrite the verify call to always return `CBL_OK`. Any local-only enforcement is defeatable. cblicense doesn't claim DRM-grade resistance.

**Mitigation:** sign the firmware itself (Pi: rauc / verified boot; ESP32: secure boot V2). Then the attacker needs the firmware-signing key to ship a tampered build. cblicense complements but doesn't replace firmware signing.

## Operational guidance

### Salt management

- 32 random bytes per product family.
- Generate ONCE with `head -c 32 /dev/urandom | xxd -p -c 64` and store it in your password manager / Vault / KMS.
- **Do not commit the salt to git.** Use a CI secret + a build-time injection step (e.g. `xxd -i salt.bin > generated_salt.h` during the release build).
- Rotate the salt only when you migrate to v0.2 — pre-rotation codes will then need re-issuance.

### Per-deployment / per-customer salts

If you sell to large enterprise customers who insist on isolation, use separate salts per customer family ID. Then a leak from one customer's fleet doesn't affect another's.

### Public key is fine to expose (when v0.2 lands)

Once we ship the Ed25519 mode, the public key in firmware is **not a secret**. You can print it on the kiosk, embed it in support docs, anything — only the *private* key (held by your CI / vendor portal) can sign new licenses.

## Suggested risk-tier mapping

| Product price / risk | Suggested mode |
|----------------------|---------------|
| Free / $0 | Don't bother licensing. |
| ≤ $1,000 unit | v0.1 HMAC short-codes are fine. |
| $1k – $10k unit, no recurring fees | v0.2 Ed25519 signed licenses (when shipped). |
| $10k+ unit, or any recurring/expiring SKU | v0.2 + v0.5 hardware-rooted. |
| Safety-critical / regulated (e.g. medical, automotive) | None of this is appropriate; talk to a real cryptographer. |
